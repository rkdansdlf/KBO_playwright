# KBO Play-by-Play (PBP) 속도 및 안정성 개선 계획

## 1. 개요
본 계획은 KBO 문자중계(PBP) 데이터 수집 파이프라인의 **실시간 수집 속도(Speed)**와 **데이터 무결성(Stability)**을 동시에 극대화하기 위한 전략을 정의합니다. 주 수집원인 네이버 스포츠 API를 최대한 활용하면서, KBO 공식 사이트를 견고한 폴백(Fallback)으로 구성합니다.

## 2. 핵심 목표
*   **속도 (Speed):** 라이브 경기 중 지연 시간을 최소화하여 실시간 분석 및 서비스 제공에 차질이 없도록 합니다.
*   **안정성 (Stability):** 더블헤더, 서스펜디드 게임 등 예외 상황에서도 100% 정확한 데이터 매칭과 누락 없는 수집을 보장합니다.

## 3. 세부 개선 전략

### 3.1. [속도] 라이브 폴링(Polling) 최적화
현재 단일 주기로 데이터를 가져오는 방식을 경기 상황에 맞춰 동적으로 조절합니다.
*   **Dynamic Polling:**
    *   **이닝 진행 중:** 짧은 주기 (예: 5~10초)로 API 호출.
    *   **이닝 교대 및 우천 지연:** 긴 주기 (예: 30~60초)로 API 호출하여 리소스 낭비 및 차단 방지.
    *   **경기 종료 후:** 즉시 수집 중단 및 최종 검증 단계로 전환.
*   **경량화된 파싱:** 라이브 중에는 DB 쓰기 작업과 파싱을 분리하여 병목을 줄이고, 인메모리 큐를 활용하여 비동기 처리 속도를 높입니다.

### 3.2. [안정성] ID 매칭 알고리즘 고도화
KBO의 게임 ID(예: `20240510SSHH0`)와 네이버의 게임 ID 형식이 달라 발생하는 매칭 실패를 근본적으로 해결합니다.
*   **다중 조건 매칭 (Multi-factor Matching):** 단순히 팀 코드와 날짜만 비교하는 것을 넘어, 구장 정보, 시작 시간, 더블헤더 번호를 종합적으로 채점하여 가장 확률이 높은 네이버 ID를 역추적합니다.
*   **팀 코드 하드코딩 탈피:** `KBO_TO_NAVER_TEAM_CODE` 매핑의 엣지 케이스(예: 올스타전, 국가대표팀)를 정규화하는 전처리기를 도입합니다.

### 3.3. [안정성] 엄격한 정합성 검증 (Validation)
수집된 데이터가 실제 경기 결과와 정확히 일치하는지 자동으로 검증합니다.
*   **Final Score Validation:** 수집된 PBP 데이터의 마지막 타석 점수와 KBO 공식 박스스코어의 최종 점수를 비교합니다.
    *   *불일치 시:* 해당 PBP 데이터를 'Unverified'로 마킹하고, 다른 소스(KBO 사이트 등)에서 전체 재수집(Full Re-crawl)을 트리거합니다.
*   **Missing Inning Detection:** 1회부터 9회(혹은 연장)까지 이닝별 시작/종료 이벤트가 순차적으로 존재하는지 검사하여 누락된 이닝을 찾아냅니다.

### 3.4. [안정성 & 자동화] 자가 복구 파이프라인 (Auto-Healing)
일일 배치 작업(`run_daily_update.py`)의 복구 능력을 강화합니다.
*   **전수 누락 스캐너:** 매일 새벽, 전날 경기뿐만 아니라 최근 30일 내의 모든 경기를 스캔하여 PBP 데이터 누락 비율을 체크합니다.
*   **교차 폴백 (Cross-Fallback):** 네이버 API 수집에 3회 이상 실패하거나 데이터가 오염된 것으로 판단되면, 즉각 KBO `LiveText.aspx` 크롤러로 전환하여 데이터를 메웁니다.

## 4. 실행 로드맵 (Action Items)
1.  **Phase 1 (기반 작업):** `RelayCrawler`의 네이버 ID 매칭 로직 개선 및 테스트 코드 작성. (특히 더블헤더 상황)
2.  **Phase 2 (속도 향상):** 실시간 수집 스크립트(`live_crawler.py` 등)에 Dynamic Polling 적용.
3.  **Phase 3 (검증 강화):** `fetch_kbo_pbp.py`에 `--validate-final-score` 옵션 강제 적용 및 재수집 로직 구현.
4.  **Phase 4 (모니터링):** 슬랙(Slack) 알림을 통해 PBP 수집 실패 및 복구 결과를 매일 아침 리포팅.
